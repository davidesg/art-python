## Summary

`_forecast_table` — the Markdown forecast table returned by `generate_forecast` and
`update_and_forecast`, i.e. the numbers the analyst and the LLM actually read — builds
the interval as `level ± 1.96·level_std`, treating `level_std` as a standard error in
**index points**. For a Box-Cox `lam=0` (log) model, which is every CPI model in this
workflow, `fue` returns `level_std` as a **relative** standard error (a fraction of the
level): `fue`'s own HTML report multiplies it by 100 and prints it in a column headed
`Std (%)`. So the reported band is narrower than the true one by a factor of the level
itself, ≈ 100 for an index near 100.

FR_CPI at h=1: the table shows `[103.9738, 103.9811]`, a ±0.0037 band around 103.98,
when σ̂ₐ = 0.185% implies **±0.377** index points. The interval reads as if the forecast
were accurate to a ten-thousandth of an index point.

## Impact

Display/interpretation only — the point forecast, the fuf file and any downstream
comparison (e.g. `sps/forecast_compare.py`) are unaffected. But it is the table an
analyst reads to judge forecast uncertainty, and it makes every forecast look
essentially deterministic; it also silently contradicts the HTML report generated by
the same call, which shows the correct `Std (%)`.

Only `lam != 1` models are affected. For `lam = 1` (no transform) `level_std` is
already in level units and the current formula is right — which is presumably why it
was written this way.

## Reproduction

```python
import numpy as np, fue
_, m = fue.inp.load("FR_CPI_S.pre")   # FR HICP, log model (boxlam=0), n=216
m.fit()
fr = m.forecast_fuf(24)

lvl = float(fr.level[0]); se = float(fr.level_std[0])
sa  = float(np.sqrt(m._result.sigma2)) / m.refactor        # 0.001848 (log units)

print(f"level_std[0]      = {se:.6f}")                     # 0.001848  <- relative
print(f"ART band          = +-{1.96*se:.4f}")              # +-0.0036
print(f"correct band      = +-{1.96*se*lvl:.4f}")          # +-0.3766
print(f"ratio             =  {lvl:.1f}x")                  # ~104
```

Against the HTML report written by the same `generate_forecast` call (correct):

```
    1/2020 |   103.98 |     0.18        <- Level, Std (%)
```

`0.18 %` of 103.98 = 0.19 index points ⇒ 95 % band ±0.377. The Markdown table for that
same row says `[103.9738, 103.9811]`.

## Root cause

`src/art/mcp_server.py:3068`:

```python
        se   = float(fr.level_std[h])
        lo, hi = lvl - 1.96 * se, lvl + 1.96 * se
```

`fue/forecast.py:531` computes `level_std = np.sqrt(v1) / refactor`, i.e. the standard
error **in the transformed (log) space, un-rescaled** — a relative error for `lam=0`.
The docstring of `_forecast_table` repeats the wrong reading ("`.level_std` (s.e.),
95% band = level ± 1.96·se"); the comment in `fue`'s own consumer states the right one
("level_std is refactor-invariant; *100 gives %").

## Fix

Transform the band, not the standard error — for `lam = 0` the interval is
multiplicative:

```python
        se = float(fr.level_std[h])
        if boxlam == 0.0:                                   # log: relative s.e.
            lo, hi = lvl * np.exp(-1.96 * se), lvl * np.exp(1.96 * se)
        else:
            lo, hi = lvl - 1.96 * se, lvl + 1.96 * se
```

`_forecast_table(ts, fr, horizon)` currently receives no model, so `boxlam` has to be
passed in (both call sites have the model at hand). General Box-Cox would apply the
inverse transform to the band endpoints computed in the transformed space — the same
thing for `lam=0`, and correct for any `lam`.

Fix the `_forecast_table` docstring too: it is what propagated the wrong reading.

## Validation

- Assert the h=1 band half-width equals `1.96 · σ̂ₐ · level` for a log model (FR_CPI:
  ±0.377, not ±0.0037).
- Assert the Markdown table and the HTML report of the same `generate_forecast` call
  agree: `(hi − lo) / 2 / level ≈ 1.96 · Std(%) / 100`.
- Add a `lam=1` case to check the additive branch is untouched.
